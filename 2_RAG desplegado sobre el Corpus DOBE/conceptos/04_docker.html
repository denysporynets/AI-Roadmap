Empaquetar en una caja: Docker

imagen · contenedor · capas · build context · 29 jun 2026 · protocolo ds-nexus
En una frase: meter el sistema operativo, Python, las librerías, tu código, el corpus y el índice en una caja sellada (la imagen) que corre igual en tu Mac y en la nube — para que "en mi máquina funcionaba" deje de ser una excusa.

El problema que resuelve

Hoy tu app corre gracias a tu Mac: tu Python 3.13, tu .venv, tus librerías en sus versiones. Si la llevas a otro ordenador o a la nube, nada garantiza que esté lo mismo instalado → rompe. Docker empaqueta el entorno entero junto al código, así que el destino no tiene que preparar nada.

La mudanza. En vez de pedirle a la casa nueva que tenga tus muebles, mandas un contenedor de mudanza con todo dentro, ya montado. Llega, lo abres, y funciona idéntico — no dependes de lo que hubiera (o faltara) en destino.

Imagen vs contenedor = receta vs plato. La imagen es la receta congelada (siempre igual); el contenedor es el plato cocinado a partir de ella. De una imagen puedes arrancar muchos contenedores idénticos.

Las dos decisiones de oficio del Dockerfile

1 · Dependencias ANTES que el código (cacheo por capas)

Docker construye la imagen en capas apiladas y cachea cada una. Si una capa no cambia, la reutiliza. Por eso copiamos e instalamos requirements.txt antes de copiar el código: así, cuando cambias api.py, no se reinstalan las librerías otra vez.

COPY requirements.txt .          # capa 1 — cambia poco
RUN  pip install -r requirements.txt  # capa 2 — se CACHEA (lo caro)
COPY rag_core.py api.py ./       # capa 3 — cambia a menudo (lo barato)
Mise en place. El cocinero deja preparados los ingredientes que casi nunca cambian (capa cacheada) y solo rehace el emplatado final cada vez. Si invirtieras el orden, picarías toda la cebolla en cada comanda.

2 · Copiar solo lo mínimo, nombrado (lista blanca)

En vez de COPY . . (mete todo), nombramos uno a uno los ficheros que entran: motor, API, frontend, corpus e índice. Doble ventaja: la imagen es ligera y el .env no puede colarse aunque exista, porque no lo nombramos.

COPY rag_core.py api.py ./
COPY static/ ./static/
COPY corpus_dobe_enriquecido.json ./
COPY embeddings/indice_a.npy embeddings/indice_a.json ./embeddings/

El secreto que NO viaja en la caja

La imagen lleva el corpus (la necesita para responder), pero nunca las claves. La OPENAI_API_KEY y el APP_TOKEN llegan por entorno en tiempo de ejecución (en la nube los inyecta Secret Manager, Fase 5). Por eso rag_core.py y api.py leen os.environ primero: el mismo código corre en tu Mac (lee del .env) y en Cloud Run (lee del secreto) sin tocar nada.

Por qué la imagen sí lleva el corpus. Es tu proceso personal — pero va dentro de una caja que se guarda en un registro privado (Artifact Registry) y cuya puerta de consulta exige token. Privacidad por capas: registro privado + endpoint con token, no por esconder el dato.

El .dockerignore y un truco: la negación

Recorta el contexto y blinda secretos. Aquí, una regla fina: queríamos fuera todos los .json de embeddings/ (metadatos personales) menos el índice servible. Se hace excluyendo y luego re-incluyendo con ! (en Docker gana el último patrón que coincide):

embeddings/*.json            # fuera meta_g1, meta_seg…
!embeddings/indice_a.json    # …pero este SÍ (la '!' reincluye)

Detalle sutil: *.html en la raíz no alcanza a static/index.html, porque en Docker el * no cruza la barra /. Por eso el frontend sobrevive.

Cómo se verificó (sin Docker instalado)

El plan construye en la nube (Cloud Build), así que no hacía falta Docker local. Para comprobar la receta antes de gastar un build, se simuló el filtrado del .dockerignore con las reglas reales de Docker y se auditó qué entraría:

Para el CV (cómo contarlo)

Contenericé un servicio FastAPI con un Dockerfile sobre python:3.13-slim, ordenando las capas para aprovechar la caché (dependencias antes que código) y usando COPY de lista blanca + .dockerignore (con reglas de negación) para una imagen mínima y sin secretos: las claves se inyectan por entorno (compatible con Secret Manager), el contexto de build se redujo a lo imprescindible y se auditó antes de construir.
