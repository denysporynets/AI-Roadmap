A producción: Cloud Run

serverless · escala a cero · secret manager · 29 jun 2026 · protocolo ds-nexus
En una frase: subir la caja (la imagen Docker) a un servicio que la arranca solo cuando llega una visita, le da una URL pública y le inyecta las claves desde una bóveda — sin servidores que mantener y pagando casi nada.

Qué es "serverless" (y por qué no significa "sin servidores")

Sí hay servidores; lo que no haces es gestionarlos. No eliges máquina, ni instalas SO, ni parcheas, ni la enciendes/apagas. Subes el contenedor y Cloud Run decide cuántas copias levantar según el tráfico — incluida cero cuando no hay nadie.

El taxi vs el coche propio. Un servidor clásico es un coche: lo compras, lo aparcas, lo mantienes, pague o no lo uses. Serverless es un taxi: aparece cuando lo llamas, pagas solo el trayecto, y desaparece al bajarte. Sin garaje, sin ITV, sin seguro.

Escala a cero: por qué esto casi no cuesta

Sin tráfico, Cloud Run mantiene 0 instancias → 0 coste de cómputo. Cuando llega la primera petición, levanta una instancia (el cold start: arrancar el contenedor + cargar tu motor en el lifespan). Por eso la primera consulta tras un rato tarda un poco más; las siguientes van calientes.

El viaje de un deploy (un solo comando)

gcloud run deploy --source .→ sube 9 ficheros→ Cloud Build (Dockerfile)→ Artifact Registry→ Cloud Run + secretos→ URL pública 
gcloud run deploy dobe-rag \
  --source . \                          # construye desde el Dockerfile
  --region europe-southwest1 \          # Madrid
  --allow-unauthenticated \             # red abierta para CARGAR la web…
  --set-secrets OPENAI_API_KEY=OPENAI_API_KEY:latest,\
                APP_TOKEN=APP_TOKEN:latest   # …pero /preguntar exige token

Dos capas de acceso (importante no confundirlas)

CapaQuién decideEn la demo
Red (--allow-unauthenticated)Cloud Run / IAMAbierta: cualquiera puede cargar la web
App (X-Token)Tu código (api.py)Cerrada: preguntar exige el token

Esto es justo "demo real protegida por token": la enseñas a quien quieras (URL viva) pero solo quien tiene la clave consulta tu corpus. La privacidad va por capas, no por esconder la URL.

El secreto, de punta a punta

La OPENAI_API_KEY nunca tocó la imagen ni el repo: se guardó en Secret Manager y Cloud Run la inyecta como variable de entorno al arrancar. Tu código (os.environ primero) la encuentra ahí. Además, la cuenta de servicio del contenedor necesitó permiso explícito (roles/secretmanager.secretAccessor) para leerla — el robot solo accede a lo justo.

El mismo código, dos casas. En tu Mac lee la clave del .env; en la nube la lee del secreto. No cambiamos ni una línea: esa portabilidad la dejamos preparada desde la Fase 1.

Para el CV (cómo contarlo)

Desplegué un servicio RAG como contenedor serverless en Google Cloud Run (build en la nube con Cloud Build, imagen en Artifact Registry privado), con secretos en Secret Manager inyectados en runtime y una cuenta de servicio con permisos mínimos. Acceso en dos capas (red pública + token de aplicación), escala a cero (coste ≈ 0 en reposo) y URL pública verificada de punta a punta. Pipeline de un solo comando, sin Docker local.
