La cara visible: el frontend

cliente · fetch · localStorage · las tres capas · 29 jun 2026 · protocolo ds-nexus
En una frase: una sola página (static/index.html) que corre en tu navegador, le pide cosas a la API con fetch() y pinta el JSON que recibe. No sabe nada de RAG: solo habla con la API por un contrato.

Servidor vs cliente: dónde corre cada cosa

Esta es la confusión típica (la misma que te hizo preguntar si cerrar Safari apagaba la app). Hay dos máquinas lógicas distintas, aunque hoy estén en el mismo Mac:

 Servidor (back)Cliente (front)

Dónde correen el proceso de uvicorn (puerto 8080)en la pestaña del navegador
Si cierras Safarisigue vivo (proceso aparte)se va (era el cliente)
Tiene la clave OpenAIsí (en su entorno)nunca (solo el token de acceso)
Restaurante. La cocina (servidor) sigue encendida aunque el comensal (navegador) se levante y se vaya. Son dos sitios. Por eso uvicorn seguía corriendo tras cerrar la pestaña — y por eso conviene killarlo cuando acabas.

El viaje de una pregunta

escribes + token→ fetch() POST /preguntar→ API (X-Token ok?)→ MotorRAG.preguntar()→ JSON de vuelta→ el navegador lo pinta

Fíjate: el navegador no decide nada del RAG. Manda texto, recibe un dict y lo dibuja. Toda la inteligencia está al otro lado del contrato.

Cómo se pinta el resultado

BandaQué muestra la web
🟢 claro / 🟡 posibleLa respuesta, con las citas [#57·pág115–117] resaltadas en ámbar + chips de fuentes
⚪ lejano"No guardo un recuerdo claro… no invento" + lo más cercano (no concluyente)

El frontend respeta el umbral del motor: si el dict trae respuesta:null, no se inventa nada — muestra la honestidad del sistema. La decisión de no alucinar se tomó en el motor; la web solo la refleja.

Detalles de oficio (pequeños pero importan)

Escapar el texto. El texto del usuario y del LLM se escapa antes de meterlo en la página (&lt; en vez de <) para que un <script> en una respuesta no se ejecute. Es la base de no abrir un agujero XSS.

Token solo en el navegador. Viaja en la cabecera X-Token de cada fetch; nunca se incrusta en el código de la página. La clave de OpenAI jamás toca el frontend.

Estados de la UI. Spinner mientras espera, y mensajes claros para 401 (token malo), error de API y "no contesta el servidor". Una interfaz honesta también lo es cuando algo falla.

La idea grande: tres capas, un contrato

# cada capa solo conoce a su vecina, y SOLO por el dict
MotorRAG.preguntar(texto)  ->  {banda, respuesta, fuentes, ...}   # la lógica
        |
  api.py  (envuelve el dict en HTTP, pone el token)              # el transporte
        |
index.html (fetch + pinta el dict)                               # la cara

Cambiar el color de un botón no toca el motor. Cambiar el modelo de embeddings no toca la web. Cada capa se puede reemplazar sin romper las otras mientras respete el contrato (el dict). Eso es desacoplar — y es justo lo que hace mantenible un sistema.

Para el CV (cómo contarlo)

Construí un frontend estático (HTML/CSS/JS sin build) servido por FastAPI con StaticFiles, que consume la API por fetch y persiste el token de acceso en localStorage. La UI refleja el umbral de confianza del motor (no muestra respuesta cuando el sistema no la tiene), escapa el contenido para evitar XSS y mantiene la clave del LLM fuera del cliente. Arquitectura desacoplada en tres capas (motor → API → UI) comunicadas por un contrato JSON.
