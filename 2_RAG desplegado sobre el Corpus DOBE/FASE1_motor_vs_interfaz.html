Fase 1 — El motor rag_core.py

RAG sobre el Corpus DOBE · Artefacto 2 · separar la lógica de la interfaz · 28 jun 2026

En la Fase 1 no añadimos ninguna función nueva al RAG. Hicimos algo más sutil y más profesional: reorganizar el código que ya funcionaba para que la misma lógica pueda servir a una terminal hoy y a una web mañana, sin duplicarse. A eso se le llama refactorizar, y es la diferencia entre un script y un sistema.

El punto de partida: lógica y presentación pegadas

Hasta ayer, toda la inteligencia del RAG vivía dentro de 23_responder.py, mezclada con las órdenes de imprimir en pantalla. El programa calculaba la respuesta y a la vez la pintaba en la terminal, en las mismas líneas:

# ANTES — la decisión y el print, enredados en la misma función
if banda(sim_top) == "lejano":
    print("⚪  No guardo un recuerdo claro de esto.")
    print(f"    (lo más cercano cae en sim {sim_top:.3f}...)")
    return
...
resp = client.chat.completions.create(...)   # llamada al LLM
print(f"{icono}  Respuesta...")            # y el print, ahí mismo

Funciona perfectamente… para la terminal. El problema aparece cuando quieres reutilizarlo.

¿Por qué esto es un problema?

Una página web no quiere print() en una terminal: quiere datos (un JSON) para dibujarlos en el navegador. Si la lógica está soldada a los print, la web no puede aprovecharla — habría que copiar y pegar toda la lógica en otro sitio. Y código duplicado = dos sitios donde corregir cada bug. La Fase 2 (la API web) sería imposible de hacer limpia.

La idea central: motor vs interfaz

La solución es una de las ideas más reutilizables de toda la ingeniería de software: separar el que PIENSA del que MUESTRA.

🔧 MOTOR
rag_core.py → devuelve datos → 🖥️ CLI · 23_responder.py 🌐 API web · Fase 2 …lo que venga
PiezaSu único trabajoQué NO hace
Motor (rag_core.py)Recuperar momentos, decidir la banda de confianza, llamar (o no) al LLM y devolver un diccionario.No imprime. No sabe si lo usa una terminal o una web. Le da igual.
Interfaz (23_responder.py)Pedir la pregunta y presentar el diccionario en pantalla (🟢🟡⚪ + 📎 Fuentes).No calcula nada. No sabe de embeddings ni de umbrales.

Un motor, muchas interfaces. La regla de oro: la lógica devuelve datos; la interfaz los presenta.

El corazón del cambio: devolver datos, no imprimirlos

El método preguntar() del motor termina devolviendo un diccionario en lugar de imprimir. Ese diccionario contiene la decisión tomada, lista para que cualquiera la use:

# DESPUÉS — rag_core.py · el motor decide y DEVUELVE (no imprime)
def preguntar(self, consulta: str) -> dict:
    top = self.recuperar(consulta)
    sim_top = top[0][1][0] if top else 0.0
    veredicto = banda(sim_top)

    base = {
        "consulta": consulta,
        "banda":    veredicto,
        "sim_top":  round(sim_top, 3),
        "respuesta": None,
        "fuentes":  [],
        "cercanos": cercanos,
    }

    # El UMBRAL manda: si no hay recuerdo claro, ni LLM ni invención.
    if veredicto == "lejano":
        return base          # ← devuelve la decisión, no la imprime

    # ...si hay recuerdo, llama al LLM y rellena respuesta + fuentes...
    return base

Cuando lanzas la pregunta-trampa «qué ropa me pongo para ir a una boda» (algo que no está en el corpus), el motor devuelve exactamente esto:

// El diccionario que devuelve el motor — esto es lo que consumirá la API
{
  "consulta": "qué ropa me pongo para ir a una boda",
  "banda": "lejano",
  "sim_top": 0.272,
  "respuesta": null,        // el motor decidió NO llamar al LLM
  "fuentes": [],
  "cercanos": [
    { "num": 67,  "sim": 0.272 },
    { "num": 126, "sim": 0.267 },
    { "num": 6,   "sim": 0.265 }
  ]
}

Mira el detalle importante: "respuesta": null y "fuentes": []. El motor tomó la decisión de no llamar al LLM (la similitud 0.272 está por debajo del umbral 0.40) y dejó esa decisión en los datos, no en un print. La terminal la lee y muestra el ⚪; la web la leerá y mostrará lo mismo. Una decisión, dos presentaciones. Eso es lo que antes no se podía hacer.

La interfaz, ahora, solo presenta

Con la lógica fuera, 23_responder.py adelgazó de 161 a ~70 líneas. Ya no calcula nada: recibe el diccionario y elige cómo pintarlo. Compara el antes y el después de la misma rama ⚪:

Antes · lógica + print juntos

    print("⚪  No guardo...")

El 23 de después no sabe qué es un embedding ni un umbral. Solo sabe leer un diccionario y pintarlo.

Tres conceptos técnicos que aparecen aquí

1 · Una clase que carga el estado una sola vez

El motor es una clase (MotorRAG). Su constructor (__init__) carga el índice de vectores y el corpus en el momento de crearse, y los guarda en memoria. Luego cada pregunta reutiliza eso ya cargado.

class MotorRAG:
    def __init__(self):
        self.vecs   = np.load(EMB / "indice_a.npy")   # 257 vectores, UNA vez
        self.corpus = cargar_corpus()
        self.client = OpenAI()

Ejemplo del mundo real: es como un cocinero que prepara la cocina al abrir (saca cuchillos, enciende fuegos) en vez de montar la cocina entera cada vez que entra un pedido. Abrir el archivo de 257 vectores en cada pregunta sería lento y absurdo. En la Fase 2, la API encenderá el motor una vez al arrancar y atenderá miles de preguntas con esa misma instancia ya caliente.

2 · El diccionario como contrato entre motor e interfaz

El motor y la interfaz se ponen de acuerdo en una forma fija de diccionario. Mientras el motor respete ese "contrato", puede cambiar por dentro (mejorar el retrieval, cambiar de gpt-4o-mini a Claude) sin que la interfaz se entere.

CampoQué guarda
consultaLa pregunta original.
bandaEl veredicto de confianza: claro / posible / lejano.
sim_topLa similitud del mejor momento (0–1).
respuestaEl texto del LLM, o null si no hubo recuerdo.
fuentesLos momentos citados: [{num, pages}].
cercanosLos 3 más próximos, por si quieres mirarlos a mano.

Por qué importa: ese diccionario se convierte en JSON con una sola línea en la API. El JSON es el idioma universal con el que hablan el backend y el navegador. Diseñar bien ese contrato hoy es lo que hace que la Fase 2 sea casi trivial.

3 · La configuración vive fuera del código (entorno primero, .env después)

La clave de OpenAI nunca se escribe en el código. El motor la busca primero en las variables de entorno del sistema y, solo si no está, cae al fichero .env local:

def _asegurar_clave():
    # 1) ¿está en el entorno? (así la inyecta Cloud Run / Secret Manager)
    if os.environ.get("OPENAI_API_KEY"):
        return
    # 2) si no, leerla del .env local (tu Mac, en desarrollo)
    for line in (BASE / ".env").read_text().splitlines():
        if line.startswith("OPENAI_API_KEY="):
            os.environ["OPENAI_API_KEY"] = line.split("=", 1)[1].strip()

El concepto se llama "12-factor app": el mismo código, sin tocar una línea, corre en tu Mac (lee el .env) y en la nube (lee la variable que le inyecta Google Secret Manager). El secreto y el código viajan por caminos distintos: el código es público, la clave nunca. Es el complemento natural del blindaje de privacidad de la Fase 0.
