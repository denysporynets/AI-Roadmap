La pieza diferenciadora: las 417 páginas del transcript TEA1 convertidas en conocimiento consultable. Nadie más tiene este dataset. Conecta DS-NEXUS con AI Engineering de forma literal.

Artefacto Un RAG sobre el corpus DOBE: le preguntas por tu framework y responde citando pasajes reales de las 417 páginas.
Definición de «hecho» Una pregunta sobre el corpus devuelve una respuesta apoyada en fragmentos reales, con la fuente citada. Si no puedes ver de qué pasaje sale la respuesta, no está hecho.

Privacidad resuelta por capas. El corpus es un proceso de autodescubrimiento — así que: el repo público no contiene corpus ni vectores (.gitignore), la imagen vive en registro privado, las claves van en Secret Manager y la consulta exige token. Demo real, acceso por token.

Retrieval Augmented Generation (RAG) with LangChainDataCamp

La pieza que más distingue en el mercado, y por eso va aquí y no al final. Construir un RAG es fácil; saber si funciona y detectar cuándo una versión nueva empeora es lo que separa portfolio de prototipo.

Artefacto Un script que puntúa el RAG contra un dataset de eval y avisa cuando una versión nueva empeora respecto a la anterior.
Definición de «hecho» Cambias algo del RAG, corres el harness y obtienes un número que dice si mejoró o empeoró. La evaluación a ojo no cuenta.

Herramienta interna, no desplegable. Un harness de evaluación no se sirve como demo: se ejecuta en local y su valor es la disciplina — evaluar, comparar, y una puerta que bloquea regresiones. Aquí cazó un falso positivo real del umbral del RAG antes de que llegara a producción.

LLM-as-judge que puntúa fidelidad (¿la respuesta sale de la fuente?) y relevancia
Reporte que compara versiones y marca regresiones

La intuición viene de Rex+Chef en DS-NEXUS. Esta fase le pone tool use formal: un agente que ejecuta herramientas reales, no que solo conversa. Dominio elegido: búsqueda de empleo — el agente lee una oferta, la contrasta con el CV y redacta un borrador de candidatura.

Artefacto Un agente de búsqueda de empleo con function calling: orquesta tres herramientas reales (analizar oferta · match con el CV · redactar borrador). Desplegado en Cloud Run tras un token.
Definición de «hecho» Le das una oferta, el agente decide qué herramienta llamar, la ejecuta y usa el resultado — vivo en internet. Si las herramientas son simuladas, no está hecho.

Agent loop escrito a mano con el SDK de OpenAI (function calling), sin framework que lo esconda
Tres herramientas orquestadas por el modelo, con reglas de estilo horneadas en la redacción
CLI + API (FastAPI), desplegado en Cloud Run con token y secretos en Secret Manager

Building AI Agents with LangChain / LangGraphDataCamp

Técnicas que se aplican el mismo día que las aprendes — pero aquí con disciplina: cada cambio de prompt se mide con el harness del Artefacto 3. Prompt engineering sin evaluación es superstición.

Artefacto El agente versiona sus prompts y devuelve siempre estructura válida; cada cambio queda justificado por su impacto medido en la métrica.
Definición de «hecho» Puedes señalar un cambio de prompt y enseñar el número de eval que lo justifica. «Parece mejor» no cuenta.

El eval-set salió de bugs reales. Los tres fallos que el agente cometió de verdad (confundir GCP con AWS, marcar un nivel de idioma que sí estaba en el CV, saturar de cifras) se convirtieron en casos de prueba — cada uno emparejado con un caso de control que vigila que arreglarlo no rompa lo que ya funcionaba.

Ver artefacto desplegado → ⚑ El anexo CI/CD →
Prompts sacados del código a un catálogo versionado, con v1 idéntica byte a byte como línea base
Báscula híbrida: asserts para lo comprobable y juez LLM para el estilo, con casos de bug y de control emparejados
Few-shot y chain-of-thought aplicados solo donde mueven la métrica; A/B contra la línea base
Structured outputs con esquema validado en todo el pipeline, fail-closed
La báscula corre sola en GitHub Actions: cada push pasa por la puerta, con el listón calibrado sobre el ruido medido
