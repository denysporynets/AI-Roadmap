Artefacto 5 · Mes 3–4 ✓ Desplegado · endurecido

Endurecimiento de prompts
cada cambio, un número que lo respalda

No es una app nueva: es el mismo agente del Artefacto 4, pero con disciplina. Prompts versionados, un eval-set de bugs reales con casos de control, una báscula que puntúa cada cambio, un juez-LLM para lo que ningún if mide, y structured outputs con Pydantic. Ningún prompt entra en producción a ojo.

La idea
De «a ojo» a disciplina medida

El Artefacto 4 funcionaba, pero sus prompts se afinaron a ojo: los escribí, probé una vez, y si colaba, a producción. Durante su construcción cacé tres bugs de prompt reales y los parcheé sobre la marcha metiendo frases. Funcionaban… pero nunca medí si funcionaban siempre. Este artefacto convierte ese «a ojo» en un proceso: esos tres bugs se vuelven el eval-set perfecto, porque son reales.

La regla del artefacto. Es la misma filosofía del harness del Artefacto 3, mudada a prompts: prompt engineering sin evaluación es superstición. Se toca un prompt, se mide, y solo el número decide.
El instrumento
Una báscula que mide las dos caras

El eval-set va en parejas: cada bug (un fallo real) tiene su control (el daño de pasarse de frenada al arreglarlo). El bug mide si mejoras; el control, si rompes. Sin el control, la tabla te miente con buena cara.

Cada fila guarda el hash del prompt que se usó de verdad: si alguien edita una versión publicada en vez de crear otra, el hash cambia y las corridas viejas dejan de cuadrar. La regla «una versión no se edita» pasa de comentario a comprobable.

Por qué medir bien importa
La muestra pequeña casi invierte la verdad

Al comparar la carta v1 contra la nueva v3 con un juez-LLM (un gpt-4o puntuando lo que un if no ve: si la carta suena a persona o a plantilla), con 3 cartas el juez dijo que v3 era peor. Con 15, la conclusión se dio la vuelta entera.

juez de estilo · v1 vs v3N pequeño engaña
# N = 3   → v3 parece PEOR en todo   (arranque 2.33 vs 3.33)  ✗ falso
# N = 15  → v3 claramente MEJOR       (arranque 4.20 vs 2.80)  ✓
#          y v1 abre con plantilla 7 de 15 veces; v3, 0 de 15.

Aquellas 3 cartas pillaron por azar un 1 y un 2 que en 15 tiradas ni asoman. La muestra pequeña no fue solo imprecisa: invirtió el signo de la respuesta. Casi desplegamos la conclusión contraria a la verdad — la lección más cara del artefacto.

La forma, garantizada
Structured outputs: se acabó el json.loads a ciegas

Pedirle al modelo «devuelve JSON, por favor» no garantiza qué hay dentro: si falta una clave, el fallo aparece tres funciones más abajo como un None que nadie pidió. La última pieza cambia la súplica por un molde Pydantic que el decodificador está obligado a rellenar.

herramientas.py · el contrato de formaconstriñe + valida
class Oferta(BaseModel):
    puesto: str
    empresa: str | None          # null explícito, no ausencia muda
    imprescindibles: list[str]
    ...

oferta = cliente.chat.completions.parse(
    response_format=Oferta).choices[0].message.parsed   # ya validado

Y lo bonito del cierre: cambiamos toda la fontanería de salida y re-corrimos la báscula. Como los prompts no se tocaron, la calidad tenía que aguantar — y el número lo demostró. Cambiar el cómo no cambió el qué.

Cómo está construido
Las capas de la disciplina
prompts
prompts.py — los prompts salen de estar horneados en cada función a un módulo versionado: un catálogo de versiones (v1, v2, v3) y cuál está activa. Cambiar una línea de aquí = desplegar un prompt nuevo.
eval-set
evaluacion/casos.json — los tres bugs reales del Artefacto 4, cada uno emparejado con su control. Las comprobaciones son código (asserts baratos y deterministas), no reglas en un JSON.
báscula
evaluar_prompts.py — corre el eval-set N veces por caso (temp 0 no es determinista), guarda hash + salida de cada fila y añade la corrida al historial con un cerrojo (append-only, sin carreras).
juez
juez_estilo.py — un LLM-as-judge (gpt-4o, más capaz que el generador para evitar el self-preference bias) que puntúa lo cualitativo: arranque, voz, enganche. Opt-in, porque cuesta.
forma
moldes Pydantic — structured outputs vía parse(): la salida queda constreñida al esquema y validada. Fail-closed si el modelo rehúsa: error claro, no un None mudo.
despliegue
Cloud Run — el agente endurecido re-desplegado sobre la misma imagen con lista blanca. Antes de subir, /code-review cazó una captura de token que iba a colarse al build.
Lo que se entregó
Los criterios, cumplidos
Prompts versionados y activables
Un catálogo (v1/v2/v3) con la versión activa como una línea. Cada fila del historial guarda el hash: una versión publicada no se edita, y es comprobable.
Eval-set de bugs reales con controles
Casos en parejas bug/control; el bug mide si mejoras, el control si rompes. Comprobaciones en Python, deterministas, con repeticiones porque temperatura 0 no lo es.
Mejora medida, no opinada
Numerología de 0.25 a 1.0 con few-shot + chain-of-thought; el juez-LLM confirmó el estilo. Cada versión ganó contra la anterior con un número, no con un «parece mejor».
Structured outputs con validación de esquema
Las tres herramientas devuelven moldes Pydantic constreñidos y validados; la báscula demostró que el cambio de fontanería no regresó la calidad.
Definición de "hecho" — cumplida
El mismo agente, vivo en producción, pero ahora cada cambio de prompt pasa por una báscula y ninguno entra sin un número que lo respalde. La forma de cada salida está garantizada por un esquema.

Por qué este artefacto importa

Escribir buenos prompts lo intenta cualquiera. Lo que distingue es versionarlos y medir cada cambio contra un eval-set — cazar la regresión que arregla un caso y rompe otro antes de que llegue a producción. No «tocar y rezar»: tocar y demostrar.
