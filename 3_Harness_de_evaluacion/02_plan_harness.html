Harness de evaluación del RAG DOBE

plano antes de construir · fase a fase con verificación · material PROPIO · 29 jun 2026 · actualizado 9 jul 2026 · protocolo ds-nexus
Estado (9 jul 2026): Fases 0 → 4 COMPLETADAS y verificadas. El harness está entero y funcionando como puerta de calidad local. Solo queda la Fase 5 (lucirlo) para cerrar el artefacto. Detalle fase a fase abajo.
Objetivo: montar un banco de pruebas automático que mida si tu RAG DOBE responde bien (sin alucinar, al grano) y que avise si una versión nueva empeora respecto a la anterior. Tres piezas del roadmap: dataset de evaluación + juez-LLM (faithfulness/relevancia) + detección de regresiones. Se construye contra el motor que ya tienes (MotorRAG.preguntar()), sin tocarlo.

El giro propio: tu sistema sabe callarse

Lo que te diferencia del tutorial. El RAG de DataCamp siempre responde, así que solo mide faithfulness y relevancia. El tuyo no: si el mejor recuerdo cae en ⚪ (banda="lejano"), el motor se calla y no llama al LLM. Por tanto tu harness necesita una tercera métrica que ellos no tienen: ¿acertó al callarse? Una pregunta sobre algo ausente del corpus (ej. C6 "irse de casa", que verificamos AUSENTE) debe dar ⚪. Si responde, es un falso positivo. Si una pregunta con recuerdo claro da ⚪, es un falso negativo. Esto es ingeniería tuya, y es justo el concepto nuevo que explicaremos para el CV.

Las métricas que mediremos

MétricaQué mideCómo
Acierto del umbral¿Clasificó bien claro/posible/lejano?Comparar banda devuelta vs. banda esperada del dataset (sin LLM, gratis)
Faithfulness¿La respuesta sale solo de los momentos? ¿No inventa recuerdos?Juez-LLM con rúbrica, recibe contexto + respuesta
Relevancia¿Responde a lo que se preguntó?Juez-LLM con rúbrica, recibe pregunta + respuesta
Cita fuentes¿Incluyó [#num · pág] como exige el sistema?Check barato (regex), sin LLM
Decisión consciente (no espejo): NO usaremos BLEU/ROUGE. Tu corpus es episódico, no hay "respuesta oro" única — comparar solapamiento de palabras no diría nada. Vamos con juez-LLM, que es justo lo que el tutorial recomienda para este caso.

Las fases

Fase 0 · El dataset de evaluación ✅ hecho
material PROPIO · sin coste · el cimiento de todo
  • Un dataset_eval.json: ~12–15 consultas etiquetadas a mano por ti.
  • Cada entrada: pregunta, tipo, banda_esperada y una nota de qué se espera.
  • Tipos a propósito (como el dataset-trampa del tutorial): directa (recuerdo claro literal), síntesis (combina varios momentos), disparador de alucinación (ausente del corpus → debe dar ⚪ o admitir "no encuentro"), frontera (cerca del umbral).
✓ Verificación: revisamos juntos las 12–15 preguntas y sus etiquetas antes de seguir. El dataset es lo más importante; si está mal, todo lo demás miente.
Fase 1 · Los jueces (faithfulness + relevancia) ✅ hecho
eval_jueces.py · juez = gpt-4o (más capaz que el generador gpt-4o-mini)
  • Dos funciones de juez, con rúbrica 1–5 adaptada a memoria episódica, temperature=0 y salida JSON forzada ({"score", "reason"}).
  • Juez más capaz que el generador → evita el self-preference bias (que el modelo se apruebe a sí mismo).
  • Faithfulness recibe contexto+respuesta; relevancia recibe solo pregunta+respuesta.
✓ Verificación: pasamos los jueces a mano sobre 2–3 ejemplos (uno bueno, uno alucinado) y comprobamos que las notas tienen sentido y la "reason" explica bien.
Fase 2 · El bucle de evaluación ✅ hecho
evaluar.py · recorre el dataset → produce un informe
  • Para cada pregunta: llama a MotorRAG.preguntar(), mide el acierto del umbral, y si respondió pasa los dos jueces; si se calló evalúa si debía callarse.
  • Agrega: faithfulness media, relevancia media, % de umbral correcto, % con citas.
  • Salida doble: un resultados.json (máquina) + un informe HTML legible a nuestro estilo.
✓ Verificación: corremos la evaluación completa y leemos el informe. Las preguntas-trampa deberían salir señaladas; las directas, con nota alta.
Fase 3 · Línea base + detección de regresiones ✅ hecho
comparar.py · el corazón del "harness"
  • Guardar la primera evaluación buena como línea base (baseline.json).
  • Un comparador: re-evalúa, compara con la base y avisa si algo baja más de una tolerancia (p. ej. faithfulness media cae > 0.3, o aparece un nuevo falso positivo del umbral).
✓ Verificación: provocamos una regresión a propósito (p. ej. bajar el umbral) y comprobamos que el comparador la pilla y la reporta.
Fase 4 · pytest — puerta de calidad local ✅ hecho · 9 jul
test_calidad.py · 6 tests · reutiliza comparar.py · NO gasta API
  • Alcance elegido: pytest LOCAL (GitHub Actions se aparca como mejora futura — no meter secretos ni gastar API por push; coherente con el ritmo pausado).
  • Métricas envueltas como asserts que reutilizan la lógica de comparar.py (una sola definición de "regresión"): sin regresiones duras · ninguna alucinación nueva · decisión no empeora · faithfulness/relevancia no caen > tolerancia · citas no bajan.
  • Patrón quality gate: el paso caro (evaluar.py con jueces LLM) corre UNA vez y escribe resultados.json; pytest juzga esa salida gratis. Evalúa una vez → testea cuantas quieras.
✓ Verificado: 6 verdes con la versión buena (umbral 0.43); 3 rojos con la regresión del umbral 0.40 (pilla el falso positivo de T1). En 0.02 s, sin API. Confirma el hallazgo de la Fase 3: lo que se rompe es la decisión, no las medias.

Privacidad (no repetir el susto)

El dataset de evaluación es material sensible. Contiene preguntas y respuestas esperadas sobre tu corpus personal. Igual que en la Fase 0 del Artefacto 2 blindamos el corpus, dataset_eval.json, baseline.json y resultados.json van al .gitignore antes de cualquier commit. Lo publicable será el código de los jueces y el bucle, nunca los datos.

Coste

El juez usa gpt-4o (más caro que mini) pero solo sobre ~12–15 preguntas × 2 jueces = ~30 llamadas por evaluación completa. Céntimos por corrida. Las preguntas que dan ⚪ ni generan ni se juzgan con LLM → gratis. El muestreo al 10% del tutorial no hace falta a esta escala.

Qué decidir antes de empezar — resuelto

Las dos dudas quedaron resueltas: (1) Fase 4 = pytest LOCAL (el CI/CD con GitHub Actions se aparca como mejora futura, igual que la opción C del Artefacto 2). (2) El dataset arrancó pequeño con 7 casos golden etiquetados a mano (2 directas, 1 síntesis, 2 trampas, 2 frontera), suficientes para calibrar el harness — se puede ampliar más adelante ("tengo toda una vida para afinar").
