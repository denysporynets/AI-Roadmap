La puerta de calidad

quality gate · separar lo caro de lo barato · una sola fuente de verdad · 9 jul 2026 · protocolo ds-nexus
Una puerta de calidad convierte el harness en una red de seguridad automática. El truco de ingeniería es separar el paso caro del barato: interrogar al RAG y llamar a los jueces LLM cuesta dinero y se hace una vez; comprobar si ese resultado empeoró respecto a la versión anterior es gratis y se hace tantas veces como quieras. Al final la puerta es un simple pytest que se pone verde o rojo — y un pipeline entiende ese color.

1 · El problema: "mejoré el RAG"… ¿seguro?

Tocas un umbral, cambias un prompt, subes el modelo. ¿Mejoró o empeoró? Sin una medida, la única respuesta es "a ojo, parece que va mejor" — y a ojo no se ve lo peor que puede pasar: que una versión nueva empiece a alucinar justo en las preguntas que antes contestaba callándose. La puerta de calidad existe para que ese "parece que va mejor" se convierta en un número que no admite autoengaño.

Analogía — el detector de humos cableado a la alarma. Evaluar es oler el humo (mides fidelidad, relevancia, aciertos). La puerta es el cable que conecta ese olor con la sirena: si el humo supera un límite, salta sola, sin que nadie tenga que estar mirando. Un harness sin puerta es un detector sin cablear: mide, pero no te avisa.

2 · La idea clave: caro una vez, barato mil veces

El coste no está repartido por igual. Solo el primer paso gasta API. Separarlo del resto es lo que hace que la puerta se pueda correr en cada commit sin arruinarte:

evaluar.py
● CUESTA API
Interroga al RAG con el dataset golden + jueces LLM (fidelidad, relevancia).
→ resultados.json
comparar.py
○ gratis
Diff del resultado contra la línea base. Aplica tolerancias y marca regresiones.
→ veredicto + exit code
test_calidad.py
○ gratis
Asserts de pytest sobre ese diff. Verde si nada empeoró, rojo si algo cayó.
→ 🟢 / 🔴

pytest no toca la API: solo lee dos JSON ya producidos (la línea base y el candidato). Por eso el flujo es «evalúa una vez, testea cuantas veces quieras». El paso caro corre cuando cambias algo de verdad; la puerta se ejecuta sola sobre su salida, gratis, en cada push.

3 · Una sola fuente de verdad

Un error clásico sería que test_calidad.py reimplementara su propia idea de "qué es una regresión". Entonces tendrías dos definiciones que pueden contradecirse: el informe dice "todo bien" y el test dice "roto", o al revés. Aquí el test importa y reutiliza comparar.py —las mismas tolerancias, el mismo criterio— y solo lo envuelve en asserts legibles:

import comparar # la MISMA lógica que el informe

def test_sin_regresiones_duras(diff):
    # si comparar.py marcó una regresión, esto es ROJO
    assert diff["regresiones"] == []

Es el mismo exit code 0/1 de comparar.py, pero expresado como tests que un humano lee de un vistazo y un pipeline de CI entiende sin ambigüedad. Una definición, dos caras.

4 · La puerta en acción (pruébala)

Esta es la historia real del harness. Con el umbral en 0.43 todo pasa. Pero cuando existía la regresión del umbral blando a 0.40, la pregunta T1 "irse de casa" —ausente del corpus— se colaba como respondida: una alucinación. Cambia el candidato y mira cómo la puerta se pone roja sola:

candidato a testear:

HARNESS_CAND=resultados_u040.json pytest -v  ·  apuntar la puerta a otro candidato sin tocar el motor

Para tu CV / entrevista. «Envolví las métricas de calidad del RAG en una puerta de calidad con pytest: separé el paso caro (evaluación con jueces LLM, que gasta API y corre una vez) del barato (comparación contra una línea base, gratis, en cada cambio). El test reutiliza la lógica de comparación para no tener dos definiciones de regresión, y tiene un assert dedicado a detectar alucinaciones nuevas. Verificado: con la regresión reintroducida a propósito, la puerta se pone roja y caza el falso positivo.» → esto es MLOps aplicado, no teoría.

El patrón, en una frase

Un quality gate es lo que separa "tengo un harness" de "tengo un harness que me protege". Evaluar es medir; la puerta es actuar sobre la medida sin que yo esté delante. El día que este proyecto tenga CI en GitHub Actions, esta misma puerta es la que dice "no fusiones: acabas de introducir una alucinación". Hoy corre en local; el diseño ya está listo para el pipeline.
