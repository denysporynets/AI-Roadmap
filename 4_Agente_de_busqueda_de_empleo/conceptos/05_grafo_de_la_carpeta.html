La idea que lo ordena todo

Ves los mismos ficheros, pero hay tres fronteras distintas operando a la vez sobre ellos. Confundirlas es lo que hace que "¿por qué el agente funciona si el CV no está en GitHub?" parezca un misterio. Cada lente de abajo enciende una de esas fronteras:

Las tres lentes, en una frase cada una

La misma carpeta contada tres veces. Ninguna es "la verdadera": son capas.

⚙️ Cómo funciona

El runtime. api.py es la puerta → llama a agente.py (el bucle) → que usa las 3 herramientas de herramientas.py → que hablan con OpenAI. El .env reparte las claves; cv_denys.md es el conocimiento que el agente lee.

🌐 Qué sube a GitHub

El .gitignore es el portero del repo público. Regla de oro: la lógica sube, el dato no. Suben los .py y los libros de conceptos; se quedan en tu Mac el .env y toda la carpeta datos_personales/.

🐳 Qué entra en la imagen

El deploy no viaja por GitHub. El .gcloudignore + el Dockerfile deciden qué se mete en el contenedor de Cloud Run. Aquí sí entra el CV (el agente lo necesita), pero no entran cli.py ni los conceptos/.

El nudo que se deshace: el CV cruza una frontera pero no la otra

datos_personales/cv_denys.md está blindado en .gitignore → nunca aparece en tu repo público. Y sin embargo el agente desplegado lo lee y funciona. ¿Cómo?

Porque son dos canales independientes. GitHub Pages sirve tu web desde el repo (sin el CV). Cloud Run se construye desde un upload aparte que gobierna el .gcloudignore — y ese sí deja pasar el .md hacia dentro de la imagen. Dato privado en producción, sin pasar jamás por el repo público. Es exactamente el patrón que ya usaste en el Artefacto 2 (el corpus DOBE viajó igual).
