01 De "ejecutar y morir" a conversar

Hasta ahora, cada prueba era: editas el if __name__ == "__main__", lanzas python agente.py, imprime una vez y muere. Para probar otra oferta, repites todo.

Un REPL (Read–Eval–Print Loop: leer–evaluar–imprimir–repetir) le da vida continua. Es el patrón de la consola de Python, de psql, de cualquier chat de terminal: un bucle que espera tu entrada, la procesa, muestra el resultado y vuelve a esperar.

script de un solo uso REPL (chat de terminal) ───────────────────── ─────────────────────── lanzar lanzar │ │ procesar 1 vez ┌──▶ leer lo que escribes │ │ │ morir │ procesar (el agente gira) │ │ │ imprimir resultado └────── volver a esperar (hasta /salir)

02 Dos bucles anidados

Este es el clic mental de la fase. Ahora hay dos bucles, uno dentro del otro, y no son lo mismo:

┌─ BUCLE EXTERNO · el CLI (turnos HUMANOS) ────────────────┐ │ esperas → pegas una oferta → ves la respuesta → repites │ │ │ │ ┌─ BUCLE INTERNO · el agente (turnos del MODELO) ──┐ │ │ │ analizar_oferta → match_cv → redactar → fin │ │ │ │ (esto es la Fase 2, gira solo por dentro) │ │ │ └───────────────────────────────────────────────── ┘ │ │ │ └───────────────────────────────────────────────────────────┘

La frase para una entrevista: «El bucle del agente decide qué herramienta usar; el bucle del CLI decide cuándo volver a escuchar al humano. Son dos responsabilidades distintas — por eso viven en dos ficheros distintos.»

03 Leer una oferta multilínea

Un detalle práctico que muerde: input() lee una sola línea. Pero una oferta de LinkedIn son 20 líneas. Si no lo resolvemos, solo capturaríamos el primer renglón. La solución clásica: seguir leyendo líneas hasta una línea en blanco (o un comando), y unirlas.

def leer_oferta() -> str:
    """Lee texto de varias líneas hasta una línea vacía (o /salir)."""
    print("\nPega la oferta y deja una línea en blanco para enviar "
          "(/salir para terminar):")
    lineas = []
    while True:
        try:
            linea = input()
        except EOFError:            # Ctrl-D → tratar como salir
            return "/salir"
        if linea.strip() == "/salir":
            return "/salir"
        if linea.strip() == "":     # línea en blanco...
            if lineas:              #   ...y ya hay contenido → enviar
                break
            continue                #   ...al principio → ignorar
        lineas.append(linea)
    return "\n".join(lineas)

Leemos el bloque entero, no un renglón. La línea en blanco es el "Enter grande" que dice «ya está, procésalo».

04 Motor vs interfaz

Aquí está la idea de arquitectura, la misma que aplicaste en el Artefacto 2 (allí separaste rag_core.py del api.py). El agente es el motor; el CLI es solo una interfaz que lo alimenta con texto y muestra su salida. No mete lógica de negocio: solo habla con el humano.

# cli.py — la interfaz NO sabe nada de herramientas ni de tool calls.
from agente import agente          # ← importa el motor entero, ya montado

def main():
    print(BIENVENIDA)
    while True:                                   # bucle EXTERNO (humano)
        entrada = leer_oferta()
        if entrada.strip() == "/salir":
            print("¡Hasta luego! 👋"); break
        respuesta = agente(                        # ← delega TODO al motor
            f"Analiza esta oferta y prepárame la candidatura:\n{entrada}"
        )
        print(respuesta)

Por qué esto es oro:

El mismo agente() que llama este CLI lo llamará la API web en la Fase 4, sin tocar una línea del motor. Un motor, muchas interfaces: terminal hoy, HTTP mañana. Es exactamente lo que hiciste en el Art.2 (rag_core servía al CLI y a FastAPI). Reutilizas el patrón que ya dominas.

05 ¿"Streaming"?

En un chat normal (una sola llamada al LLM) mola ver la respuesta aparecer palabra a palabra: eso es streaming — pides stream=True e imprimes los trocitos según llegan. Pero un agente es distinto: por dentro hace varias llamadas y la mayoría no son texto, son peticiones de herramienta. Hacer streaming de eso es enredado (los tool_calls llegan a cachos).

Decisión honesta para v1: en vez de streamear tokens, mostramos los pasos. El modo verboso que ya escribimos en la Fase 2 imprime turno 1: analizar_oferta ✓… — y para un agente eso es mejor señal de vida que el texto goteando: ves qué está haciendo, no solo que está escribiendo. El streaming de tokens del texto final queda como mejora futura documentada.

$ python cli.py Pega la oferta y deja una línea en blanco para enviar… Buscamos AI Engineer (Mid) en Madrid… Imprescindible: Python, RAG, Docker… ⏎ ── turno 1: el modelo pide → analizar_oferta ✓ analizar_oferta → {"puesto":"AI Engineer",…} ── turno 2: el modelo pide → match_cv ✓ match_cv → {"encaje_global":"medio",…} ── turno 3: el modelo pide → redactar_borrador ✓ redactar_borrador → {"asunto":"Candidatura…",…} ── turno 4: respuesta final Encajas bien: cumples los imprescindibles. Aquí tu borrador…

Los pasos hacen de barra de progreso. Sabes que analizó, comparó y redactó — no un spinner mudo.

06 UX mínima

Un CLI amable no necesita mucho, pero sí lo básico para no frustrar:

  • Un saludo que explique qué hacer (pegar oferta + línea en blanco).
  • /salir para terminar con elegancia, y Ctrl-D / Ctrl-C capturados para que cerrar no escupa un traceback feo.
  • Separadores visuales entre tu entrada y la respuesta, para que no se mezclen.
  • Mensajes honestos: si algo falla o se agota max_turnos, decirlo claro (no fingir éxito).

Alcance de v1 (decisión deliberada, ritmo pausado): cada oferta es una consulta independiente — el agente arranca fresco cada vez. No mantenemos conversación entre ofertas. Los follow-ups tipo «hazme la carta más corta» exigirían conservar la memoria entre turnos humanos; queda anotado como la evolución natural, para no meter dos conceptos nuevos a la vez.
