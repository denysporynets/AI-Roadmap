Una API con portero

01 Las dos piezas de esta fase

El principio es el mismo que ya usaste en el Artefacto 2 (RAG): un motor, muchas interfaces. El motor no cambia; solo le añadimos una boca nueva.

┌──────────────────────────────────────┐ │ MOTOR · agente.py (no se toca) │ │ procesar_oferta(texto) → dict │ └──────┬─────────────────────┬──────────┘ │ │ ┌──────▼──────┐ ┌─────▼───────────┐ │ cli.py │ │ api.py │ ← pieza nueva │ (terminal) │ │ (HTTP + token) │ └─────────────┘ └─────────────────┘

02 El motor ahora devuelve datos

El CLI se contentaba con un texto. Pero una API que alimenta a una web quiere estructura: no solo la respuesta, también qué pasos dio el agente, cuántos turnos y si terminó bien. Por eso refactorizamos el bucle para que devuelva un dict — igual que rag_core.py devolvía un dict en el Art.2.

# agente.py — el bucle (ahora núcleo reutilizable) devuelve estructura
def _correr_bucle(peticion, max_turnos=6, verboso=True) -> dict:
    ...
    if not msg.tool_calls:                       # terminó
        return {"respuesta": msg.content, "pasos": pasos,
                "turnos": turno, "completado": True}
    ...

def agente(peticion, ...) -> str:                # envoltorio para el CLI
    return _correr_bucle(peticion, ...)["respuesta"]

def procesar_oferta(texto, max_turnos=6) -> dict:  # envoltorio para la API
    peticion = f"Analiza esta oferta y prepárame la candidatura:\n{texto}"
    return _correr_bucle(peticion, max_turnos, verboso=False)

Una sola definición del bucle, dos puertas de salida. El CLI toma ["respuesta"]; la API toma el dict entero. Ese pasos es la traza que la web pintará como barra de progreso.

03 Qué es una API HTTP

Ya montaste una en el Art.2, así que esto es memoria muscular. Una API HTTP es un programa que escucha peticiones por la red y devuelve respuestas. Aquí, con FastAPI, definimos dos rutas y una sonda:

RutaMétodoTokenQué hace
/saludGETno¿está vivo? Para monitorización (Cloud Run la usa)
/preguntarPOSTsírecibe {"oferta": "..."} y corre el agente
/GETnosirve el frontend (Fase 6)

La oferta viaja en el cuerpo (body) de un POST como JSON, y el token en una cabecera (header) llamada X-Token. FastAPI valida la forma del body con un modelo Pydantic — si falta oferta, rechaza solo.

04 El token como portero

Este es el corazón de la fase. Cada llamada al agente cuesta ~4 peticiones a OpenAI (el bucle), así que dejarlo abierto sería regalar tu dinero a cualquier bot. El token es un portero: si no coincide con APP_TOKEN, la respuesta es 401 y —clave— el agente ni se ejecuta.

@app.post("/preguntar")
def preguntar(peticion: Peticion, x_token: str = Header(default="")):
    if x_token != APP_TOKEN or not APP_TOKEN:        # portero
        raise HTTPException(status_code=401, detail="Token inválido o ausente.")
    if not peticion.oferta.strip():
        raise HTTPException(status_code=400, detail="La oferta está vacía.")
    return procesar_oferta(peticion.oferta)          # ← solo se llega aquí con token OK

Fail-closed: fíjate en or not APP_TOKEN. Si por un despiste el token quedara vacío, la puerta se cierra para todos en vez de abrirse para todos. En seguridad, el fallo por defecto debe ser "denegar", nunca "permitir". Es una decisión de una línea que separa un juguete de algo serio.

05 lifespan: preparar una vez

Arrancar el servidor por cada petición sería absurdo. FastAPI tiene lifespan: un bloque que corre una vez al levantar el servicio (para "calentar" el motor: crear el cliente OpenAI, cachear el CV) y otra al apagarlo. En el Art.2 cargabas ahí el índice del RAG; aquí basta con avisar de que el token está puesto.

@asynccontextmanager
async def lifespan(app):
    if not APP_TOKEN:
        print("⚠️  APP_TOKEN vacío: la API rechazará TODO (fail-closed).")
    print("✅ Agente listo. Esperando peticiones.")
    yield          # ← aquí el servidor vive y atiende; lo de abajo corre al apagar

07 El ejemplo real, paso a paso ✦

Esto es lo que salió al arrancar la API en local (uvicorn api:app --port 8099) y llamarla con curl. Las cuatro pruebas, tal cual:

Prueba 1 · /salud sin token → vivo

$ curl -s http://127.0.0.1:8099/salud {"estado":"ok","protegido":true} # vivo, y con token configurado

Pruebas 2 y 3 · el portero rechaza 401

# sin cabecera de token $ curl -X POST .../preguntar -d '{"oferta":"Data Scientist Python"}' HTTP 401 # con un token equivocado $ curl -X POST .../preguntar -H "X-Token: malo" -d '{"oferta":"..."}' HTTP 401 # el agente NI se ejecutó → 0 gasto de API

Prueba 4 · token correcto 200 → el agente gira

$ curl -X POST .../preguntar \ -H "X-Token: ****" \ -d '{"oferta":"Buscamos AI Engineer Mid en Madrid. Imprescindible: Python, RAG, Docker, GCP o AWS, ingles B2. Se valora FastAPI."}' turnos: 4 pasos: ['analizar_oferta', 'match_cv', 'redactar_borrador'] completado: True --- respuesta --- He analizado la oferta para el puesto de AI Engineer Mid... Imprescindibles: Python, RAG, Docker, Cloud (GCP o AWS), Inglés B2 Encaje Global: Medio ... (+ borrador de candidatura)

El campo pasos deja ver el bucle uno a uno — el agente decidió, en este orden:

1
analizar_oferta
texto crudo → requisitos estructurados (imprescindibles, deseables, modalidad)
2
match_cv
esos requisitos vs tu CV → encaje "Medio", qué cumples, qué falta
3
redactar_borrador
oferta + encaje → borrador de candidatura (reglas de estilo horneadas)
4
respuesta final (sin herramientas)
el modelo redacta el resumen para ti y el bucle para → completado: True

Cuatro turnos, tres herramientas, orden elegido por el modelo. Exactamente la misma danza del CLI, ahora servida por HTTP y devuelta como JSON para que una web la pueda pintar.

08 Secretos: hoy y al desplegar

El token y la API key son secretos: nunca en el código, nunca en git. Hoy viven en .env (blindado en .gitignore). Al desplegar en la Fase 5 pasarán a Secret Manager de GCP, exactamente como en el Art.2. El código no cambia: os.environ los lee igual venga de un fichero o de la nube.

Recordatorio para el deploy: el CV (cv_denys.md) no es un secreto de entorno, es un fichero — y se horneará dentro de la imagen Docker. Revisaremos .gcloudignore antes del push, como acordamos.
