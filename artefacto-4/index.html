Agente de búsqueda de empleo
tool use real, escrito a mano

Un agente que lee una oferta, la contrasta con mi CV y redacta un borrador de candidatura — orquestando tres herramientas en un bucle de function calling que escribí a mano sobre el SDK de OpenAI, sin LangGraph. Desplegado en Cloud Run, protegido por token. Cuarto artefacto de la ruta Data Science → AI Engineering.

Qué hace
Del RAG de un paso al agente de un bucle

El RAG del Artefacto 2 es de un solo salto: pregunta → recupera → responde. Un agente es distinto: tiene varias herramientas y un bucle que decide cuál usar, en qué orden y cuándo ha terminado. El camino no está fijado en el código — lo traza el modelo, leyendo el resultado de cada paso.

Paso 1
analizar_oferta
texto crudo → requisitos
→
Paso 2
match_cv
requisitos vs CV → encaje
→
Paso 3
redactar_borrador
encaje → carta de candidatura

El modelo elige ese orden solo. Tú solo pegas la oferta; el bucle piensa, actúa (llama una herramienta), observa el resultado y repite hasta entregar la respuesta.

La decisión de diseño
El bucle, a mano — no una caja opaca

Podría haber usado LangGraph. Elegí escribir el agent loop yo mismo sobre el SDK de OpenAI: esquemas de herramientas, despachador y guardarraíles. El motivo es de criterio, no de nostalgia — quiero entender la mecánica que los frameworks abstraen, y el bucle entero cabe en una pantalla.

agente.py · el corazón del agentefunction calling puro
for turno in range(1, max_turnos + 1):   # guardarraíl: nunca infinito
    resp = cliente.chat.completions.create(
        model=MODELO, messages=mensajes, tools=TOOLS)   # PIENSA
    msg = resp.choices[0].message
    mensajes.append(msg)

    if not msg.tool_calls:        # ¿no pide herramientas? → terminó
        return msg.content

    for tc in msg.tool_calls:      # ACTÚA + OBSERVA
        mensajes.append(ejecutar_herramienta(tc))
Por qué probar en serio importa
La interfaz cazó un error de criterio

Al probar el CLI con una oferta nueva (no la de ejemplo), el agente marcó "Inglés B2" como requisito que faltaba… teniendo el CV un C1. Comparaba los niveles de forma literal, sin entender que el marco europeo está ordenado (A1 < A2 < B1 < B2 < C1 < C2) y que un nivel superior cumple uno inferior.

la regla, horneada en la herramienta match_cvfix de una frase
# IDIOMAS — marco CEFR, niveles ORDENADOS A1<A2<B1<B2<C1<C2.
# Un nivel IGUAL O SUPERIOR cumple el requisito
# (un CV con C1 CUMPLE una oferta que pide B2). nativo/bilingüe = C2.

# antes:  encaje "medio", "Inglés B2" → BLOQUEANTE  ✗
# después: encaje "alto",  "Inglés B2" → cumplido    ✓

Un agente que dice "no cumples" cuando sí cumples te haría descartar buenas ofertas. Se detectó, se corrigió con una regla en el prompt y se verificó re-corriendo el mismo caso — antes de desplegar.

Cómo está construido
Un motor, varias interfaces
herramientas
herramientas.py — tres funciones (analizar_oferta, match_cv, redactar_borrador); extracción y juicio a temperatura 0, redacción a 0.7. El CV es conocimiento, cargado por context stuffing (cabe entero, sin RAG).
agente
agente.py — el bucle de function calling: menú de herramientas (JSON Schema), despachador nombre→función y guardarraíl de turnos. Devuelve la traza de pasos, no solo texto.
interfaces
cli.py + api.py — el mismo motor servido por dos bocas: un chat de terminal (REPL) y una API REST con FastAPI. Una lógica, dos puertas.
portero
token — cada consulta son ~4 llamadas al LLM; sin token válido → 401 y el agente ni se ejecuta (fail-closed). Protege la API key.
despliegue
Docker + Cloud Run — imagen con lista blanca de ficheros, secretos en Secret Manager, escala a cero. Verificado antes de subir que el CV viaja pero el .env no.

Definición de "hecho" — cumplida

Demo protegida por token. La página carga para cualquiera (prueba de que es real), pero ejecutar el agente requiere un token de acceso: cada consulta gasta llamadas al LLM y el portero evita el uso indebido de la API key. Es una decisión de coste y seguridad, no un adorno.
